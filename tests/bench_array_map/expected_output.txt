Expected JSON schema and example output for bench_array_map.py
==============================================================

JSON Schema (keys and types)
-----------------------------
{
  "benchmark":              string  -- always "bench_array_map"
  "transport":              string  -- always "BPF_ARRAY"
  "poll_count":             integer -- number of array reads performed
  "total_increments":       integer -- final counter value (total sched_switch events)
  "mean_read_latency_us":   float   -- mean time for a single array[0] read (µs)
  "p50_read_latency_us":    float   -- 50th-percentile read latency (µs)
  "p95_read_latency_us":    float   -- 95th-percentile read latency (µs)
  "p99_read_latency_us":    float   -- 99th-percentile read latency (µs)
  "stdev_read_latency_us":  float   -- standard deviation of read latency (µs)
  "min_read_latency_us":    float   -- minimum read latency (µs)
  "max_read_latency_us":    float   -- maximum read latency (µs)
  "increment_rate_per_sec": float   -- context-switch rate observed during the run
  "elapsed_sec":            float   -- actual wall-clock duration of the run
}

Notes
-----
- BPF_ARRAY read latency (mean ~3 µs) is typically much lower than BPF_HASH
  (mean ~40 µs for a map with many entries) because a single array lookup
  requires only one bpf(BPF_MAP_LOOKUP_ELEM) syscall with O(1) memory access.
- increment_rate_per_sec reflects the kernel's context-switch rate during the
  run; this varies greatly with system load.
- total_increments is the authoritative final counter; poll_count reflects the
  number of times userspace successfully read before the deadline.
- High max_read_latency_us spikes indicate OS scheduler preemption during the
  syscall — expected occasionally on a non-RT kernel.
