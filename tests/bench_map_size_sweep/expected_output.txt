Expected JSON schema and example output for bench_map_size_sweep.py
====================================================================

JSON Schema (keys and types)
-----------------------------
{
  "benchmark": string  -- always "bench_map_size_sweep"
  "transport_results": {
    "BPF_HASH": [
      {
        "map_entries":          integer -- number of entries pre-populated in the map
        "iterations":           integer -- number of full map iterations timed
        "mean_iter_us":         float   -- mean time for a full map iteration (microseconds)
        "p50_iter_us":          float   -- 50th-percentile iteration latency (µs)
        "p95_iter_us":          float   -- 95th-percentile iteration latency (µs)
        "p99_iter_us":          float   -- 99th-percentile iteration latency (µs)
        "stdev_iter_us":        float   -- population standard deviation (µs)
        "min_iter_us":          float   -- minimum observed iteration latency (µs)
        "max_iter_us":          float   -- maximum observed iteration latency (µs)
        "total_bytes_per_iter": integer -- total bytes transferred per iteration
                                           (BPF_HASH: entries × 16 B; BPF_ARRAY: entries × 12 B)
        "bytes_per_us":         float   -- effective read bandwidth (bytes/µs = MB/s)
      },
      ...  (one entry per map size)
    ],
    "BPF_ARRAY": [
      -- same structure as BPF_HASH
    ]
  }
}

Notes
-----
- mean_iter_us scales roughly linearly with map_entries for both map types,
  because each entry requires at least one bpf() syscall round-trip.
- BPF_HASH iterates via bpf_map_get_next_key + bpf_map_lookup_elem (two
  syscalls per entry); BPF_ARRAY uses bpf_map_lookup_elem only (one syscall
  per entry), so BPF_ARRAY is generally faster at the same entry count.
- A "cache-effect cliff" may appear around 5000–50000 entries where the map
  data no longer fits in the CPU L3 cache, causing mean_iter_us to grow
  faster than linearly and bytes_per_us to drop noticeably.
- total_bytes_per_iter = entries × (key_size + value_size):
    BPF_HASH:  entries × 16 B  (8 B key + 8 B value)
    BPF_ARRAY: entries × 12 B  (4 B index + 8 B value)
- bytes_per_us is the effective read bandwidth in MB/s
  (1 bytes/µs = 1 MB/s).  Higher is better; a declining trend with growing
  map_entries indicates increasing syscall amortisation overhead.
- No kernel eBPF program is loaded; all BPF maps are created and read
  entirely from userspace, isolating the userspace map-read cost.
